Technical Architecture

Caritas HARVEST

Web portal and Android app for an aid organisation tracking agricultural impact across ~20 Indonesian villages. Data is submitted from a native Android app used by village workers and viewed by organisation staff through a web dashboard. Real-time chat with image sharing connects field workers and staff.

System Overview

How it all fits together

One FastAPI backend serves both the web and Android clients through a shared REST API and WebSocket endpoint. Nginx sits in front as a reverse proxy with TLS termination, and also serves images directly.

Sync Strategy

Mobile ↔ Backend

The Android app handles two directions of data flow. Reports push up when connectivity is available; schedules pull down on launch and periodic intervals.

▲ Push

Reporter → Cloud

Reports created offline are queued in the local Room database and synced when network becomes available.

  1. 1 Report created offline, stored in Room DB sync queue
  2. 2 WorkManager job fires when network is available
  3. 3 POST /villages/{id}/reports for each queued item
  4. 4 On 200 OK → mark synced locally; on failure → retry with backoff
▼ Pull

Cloud → Reporter

Schedules managed by staff in the web app are polled on launch and every ~30 minutes.

  1. 1 App launches or periodic timer fires (~30 min)
  2. 2 GET /villages/{id}/calendar?updated_since=<ts>
  3. 3 Incremental updates stored in Room DB for offline viewing

Idempotent writes

Each report gets a client-generated UUID before sync. The backend deduplicates on this UUID to prevent double-counting on retries.

Incremental pull

updated_since timestamp filtering so the app only fetches what changed since the last sync.

WebSocket for chat

Real-time messaging uses a persistent WebSocket connection. Schedule sync remains polling-based — different cadence, different pattern.

Image uploads

Multipart upload to FastAPI, stored locally with Pillow-generated thumbnails. Nginx serves bytes directly. Migration path to S3.

Wire Protocols

REST + WebSocket

Both clients use the same REST endpoints and the same WebSocket protocol. The backend doesn't know or care which client is connected.

Image Pipeline

Upload, process & serve

Images are uploaded via multipart POST, validated for ownership, and thumbnailed in the background via Pillow. Nginx serves the actual bytes directly once FastAPI has authorized the request and issued a signed URL.

Step 1

Upload

Client sends multipart form to POST /images/upload. FastAPI validates auth, parses with python-multipart, writes original to local filesystem.

→
Step 2

Thumbnail

FastAPI BackgroundTask calls Pillow via asyncio.to_thread to generate a thumbnail. Metadata row written to images table.

→
Step 3

Serve

GET /images/:id validates membership, returns a short-lived signed URL. Nginx serves bytes directly via try_files — FastAPI never touches the file again.

Storage Migration Path

Phase 1: Local filesystem

Originals and thumbnails in a mounted volume on the VPS. Nginx serves bytes. Simple, no external dependencies.

Phase 2: S3 / DO Spaces

Swap write path to S3 SDK. Signed URLs point to S3 directly. Eliminates disk pressure on the VPS and enables CDN.

Abstraction layer

Image service interface with save(), url_for(), and delete(). Swap implementation without touching callers.

Future

Migration path to AWS

If the project outgrows a single VPS, each layer can be moved independently. The Docker Compose setup acts as a stepping stone — same containers, different orchestrator.

1. Database first

Move Postgres to RDS or DO Managed Database. Change one connection string — the app doesn't know the difference.

2. Image storage to S3

Swap the image service implementation from local filesystem to S3 / DO Spaces. Signed URLs point to S3 directly, eliminating disk pressure and enabling CDN. Nginx image serving is no longer needed.

3. Compute

Containerised with Docker Compose, so moving to ECS Fargate or Kubernetes is straightforward. ALB replaces Nginx for TLS and routing; sticky sessions or Redis pubsub needed for WebSocket at scale.

4. Monitoring

Prometheus + Loki + Grafana run anywhere, or swap for managed equivalents like CloudWatch or Grafana Cloud.
